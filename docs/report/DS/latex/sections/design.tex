\section{Design}

The system has been designed following a microservices architectural style, where each service models a specific subdomain and exclusively owns its data.

The principal objectives of the design phase have been to:
\begin{itemize}
  \item ensure service autonomy;
  \item separate services persistence management;
  \item guarantee scalability and fault isolation;
  \item preserve local consistency while enabling eventual consistency across distributed services;
  \item separate responsibilities following domain partitioning.
\end{itemize}

A key design principle adopted is data ownership per service: no service directly accesses the database of another service. All inter-service coordination occurs either synchronously or asynchronously. This separation allows the system to tolerate partial failures, avoid cross-service synchronous operation chaining and maintain loose coupling.

\subsection{Structure (Domain Entities)}

The identification of domain entities and service boundaries has been guided by principles inspired by Domain-Driven Design (DDD).

The domain has been decomposed into bounded contexts, each representing a coherent business subdomain with its own model and behavior. This approach reduces semantic ambiguity and supports independence between different subsystems.

Each bounded context corresponds to a logically autonomous service encapsulating domain entities, business rules and its persistence layer. This alignment between domain boundaries and service boundaries reduces coupling and minimizes the need for distributed transactions, favoring asynchronous coordination and eventual consistency.

Six bounded contexts are identified: \textbf{user}, \textbf{event}, \textbf{interaction}, \textbf{chat}, \textbf{ticketing} and \textbf{notification}. Additionally, some support services are used: \textbf{auth} (Keycloak), \textbf{media} and \textbf{payments} (Stripe).

\subsubsection{Core Entities}

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{../public/core-entities.png}
  \caption{Core entities and bounded context diagram}
  \label{fig:core-entities}
\end{figure}

The diagram (fig.~\ref{fig:core-entities}) shows a simplification of the design. The diagram highlights the separation between bounded contexts and the absence of direct structural dependencies across services. Cross-service relationships are expressed through entity identifiers and domain events rather than object references or shared persistence.

\paragraph{User} Represents a platform account that can assume one of two roles: Member or Organization. Attributes include identifier, authentication-related data, profile information and event preferences.

\paragraph{Event} Represents an activity created by an Organization. Its attributes include identifier, title, description and all information about the event (date, time, location).

\paragraph{Ticket} Represents a User's receipt of payment to participate in an Event. It has its own lifecycle (creation, validation or invalidation) and is modeled independently from Event to avoid coupling financial logic with event management logic.

\paragraph{Interaction} Abstract concept representing user actions, in particular: Like (User $\to$ Event), Follow (User $\to$ User), Review (User $\to$ Event) and Participation (User $\to$ Event). Interactions are separated from Event and User entities to enable event-driven propagation of user activities.

\paragraph{Conversation} Represents a communication channel between two Users. Messages are modeled as a separate entity rather than being embedded directly within the Conversation. This separation prevents unbounded data growth inside the conversation and allows message data to be managed independently. When a user requests the list of active conversations, the system only needs to access conversation metadata without loading all associated messages.

\paragraph{Notification} Represents a system-generated alert that must be sent to a specific user (e.g., new follower, new event published from a followed organization, new message/like/review received). Notifications are derived entities: they originate from domain events generated by other services.

\subsubsection{Domain Events}

In addition to static entities, the system models domain events to improve communication between services. Examples include: \texttt{UserCreated}, \texttt{EventCreated}, \texttt{EventPublished}, \texttt{TicketPurchased}, \texttt{LikeAdded}, \texttt{NewMessage}. Domain events enable asynchronous integration between bounded contexts and allow the system to maintain eventual consistency.

\subsection{Interaction}

Communication between the different components of the system follows two interaction patterns: synchronous request--response and asynchronous event-based communication.

\begin{itemize}
  \item \textbf{Synchronous interactions between frontend and backend services}: synchronous communication is primarily used for interactions initiated by users through the frontend. The client sends a request to a service and waits for a response. This approach is suitable for operations that require an immediate result, such as authentication, event browsing or ticket purchasing.

  \item \textbf{Asynchronous event-driven communication}: communication between backend services is primarily handled through asynchronous event-driven interactions. When a service completes an operation that may affect other parts of the system, it emits a domain event describing the change. Other services subscribe to the events they are interested in and react accordingly.
\end{itemize}

\paragraph{Consistency and coordination.}
Each service guarantees consistency within its own boundaries by executing state-changing operations within local transactions. After a transaction completes, the service may emit one or more domain events describing the resulting state change. Other services process these events asynchronously, updating their own state or triggering additional operations, maintaining overall coherence through eventual consistency.

\subsection{Behaviour}

All services in the system follow a common behavioral structure when handling both user requests and inter-service communication. Three main behavioral patterns can be identified:

\subsubsection{Request-Driven Operations}

User actions are processed through a request-driven workflow. A request is received by the API interface of the corresponding service and forwarded to the layer responsible for executing the domain logic.

To guarantee reliability, the \textbf{Outbox Pattern} is adopted. Instead of publishing events directly after the state update, domain events are first stored in a dedicated outbox structure within the same transaction as the database update. Once the transaction successfully completes, the events stored in the outbox are asynchronously delivered to the message broker.

The typical workflow is:
\begin{enumerate}
  \item A client request is received through the service API.
  \item The request is validated and processed (optionally making synchronous requests to other services) by the service logic.
  \item A local transaction updates the service state and records domain events in the outbox.
  \item After the transaction completes, the events are asynchronously published.
\end{enumerate}

\subsubsection{Event-Driven Processing}

Services also react to domain events generated by other services. When an event is received, the service processes it through its domain logic and may update its internal state or trigger additional events.

The typical flow is:
\begin{enumerate}
  \item A domain event is received.
  \item The service processes the event through its domain logic.
  \item If required, a local transaction updates the service state.
  \item Additional domain events may be generated.
\end{enumerate}

\subsubsection{Real-Time Notification Propagation}

Some domain events represent user actions that must be sent to other users in real time, such as a message or a new interaction. These events trigger the creation of notification entities.

The workflow is:
\begin{enumerate}
  \item A domain event is generated by a service.
  \item The notification subsystem consumes the event.
  \item A notification entity is created and stored.
  \item The notification is delivered to relevant connected clients in real-time.
\end{enumerate}

\subsection{Architecture}

\subsubsection{Architectural Style}

The system is designed according to a \textbf{microservices architecture}. The application is decomposed into a set of small, independently deployable services, each responsible for a specific bounded context: user, event, interaction, chat, ticketing and notification. Each microservice adopts the \textbf{database-per-service} pattern.

When shared information is required (e.g., basic user data such as \texttt{id}, \texttt{name} and \texttt{avatar}), it is propagated via \textbf{domain events} rather than retrieved through synchronous queries. This intentional data duplication reduces communication overhead and keeps services decoupled.

\paragraph{Inter-service communication.}
Two complementary communication styles are adopted:

\begin{itemize}
  \item \textbf{Synchronous (HTTP REST)}: used when fulfilling a request requires invoking an operation or retrieving data from another service. For example, when an event is created, the \texttt{event} service notifies \texttt{ticketing} via its internal API so that ticket-related operations can proceed right away.

  \item \textbf{Asynchronous}: used to propagate state changes across service boundaries without creating runtime dependencies. Three mechanisms are in place:
    \begin{itemize}
      \item \textbf{Domain events (RabbitMQ)}: services publish events to the message broker; subscribers consume them independently.
      \item \textbf{WebSocket (Socket.IO)}: used by the \texttt{notification} service to maintain persistent connections with clients, enabling server-side push of real-time updates.
      \item \textbf{Webhooks}: used for third-party integrations that push notifications into the system (e.g., Stripe payment callbacks delivered to the \texttt{ticketing} service).
    \end{itemize}
\end{itemize}

\paragraph{API gateway.} All external traffic flows through an API gateway implemented with \textbf{Traefik}, which acts as a single entry point and keeps the internal service topology invisible to clients.

The motivations for this architectural choice are: service independence, scalability, loose coupling, data isolation, resilience and fault isolation.

With the exception of Stripe, all components run inside a single host using Docker Compose. No service port is exposed to the public internet; the only inbound path is through the Cloudflare tunnel.

Two categories of Docker bridge networks are used: \textbf{\texttt{eventonight-network}}, shared by all application services, Traefik, RabbitMQ and Keycloak; and \textbf{\texttt{<service>-network}}, one private network per service shared exclusively between the service and its database instance. This enforces database isolation.

\subsubsection{Data Flow}

\paragraph{Synchronous request flow (HTTP REST).}
\begin{verbatim}
Browser → Cloudflare (DNS + TLS) → cloudflared → Traefik → Service → Database
\end{verbatim}

\paragraph{Asynchronous event flow (AMQP).}
\begin{verbatim}
Service → Exchange (eventonight) → Queue (per-service) → Consumer service
\end{verbatim}

RabbitMQ uses a single \textbf{topic exchange} named \texttt{eventonight}. Each consuming service has a dedicated queue bound to the exchange with a set of topic subscriptions. The exchange fans messages out to every queue whose binding pattern matches the routing key, without the publisher knowing who the consumers are.

\subsubsection{Web API}

All application services expose a \textbf{REST API}, accessible externally via Traefik at \texttt{<service>.eventonight.site}. Every protected endpoint requires a \textbf{JWT Bearer token} in the \texttt{Authorization} header, issued by Keycloak and validated independently by each service using Keycloak's public key. The \texttt{notification} service additionally exposes a \textbf{WebSocket endpoint}.

\subsubsection{Scalability and High Availability}

The current deployment runs all components on a \textbf{single host} using Docker Compose. The architecture is designed to transition to \textbf{high availability} by adopting \textbf{Docker Swarm}.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{../public/design/Swarm.png}
  \caption{Docker Swarm deployment topology}
  \label{fig:swarm}
\end{figure}

For true HA, the Swarm should have at least \textbf{three manager nodes} using the Raft consensus protocol. Stateless application services can be freely distributed across nodes. Stateful services (RabbitMQ and databases) require placement constraints to be distributed across different machines and dedicated HA strategies: MongoDB moves to a \textbf{replica set} and RabbitMQ to a \textbf{quorum queue cluster}, both requiring an odd number of members to guarantee quorum and avoid split-brain. For the \texttt{notification} service running as multiple replicas, Traefik must be configured with sticky cookies and a \textbf{Redis adapter} for Socket.IO is required to propagate messages across all instances.

\subsection{Corner Cases}

The system is designed to be \textbf{highly available} and \textbf{eventually consistent}. Each service operates independently and failures are isolated.

\paragraph{Fault detection.} Each service exposes a \texttt{/health} endpoint and declares a \texttt{healthcheck} in Docker Compose, allowing Docker to automatically restart unhealthy tasks.

\paragraph{Startup ordering and reconnection.} In a Compose deployment, services use \texttt{depends\_on} with \texttt{condition: service\_healthy}. In a Swarm deployment, each service must instead be resilient to temporary unavailability of dependencies and implement reconnection logic with exponential backoff.

\paragraph{Error signalling.} Every API endpoint returns standard HTTP status codes: \texttt{400} for validation errors, \texttt{401} for missing or invalid tokens, \texttt{403} for authorisation failures, \texttt{404} for resources not found, \texttt{409} for conflicts, \texttt{500} for unexpected server errors.

\paragraph{Component failure and recovery.} In a Compose deployment, all containers run with \texttt{restart: unless-stopped}. In a Swarm deployment, the restart policy is managed by the Swarm scheduler, which reschedules failed tasks onto healthy nodes.

\paragraph{RabbitMQ failure.} All queues are declared as durable with persistent messages. Services handle at-least-once delivery semantics by designing message handlers to be idempotent. If RabbitMQ is unavailable when a service needs to publish, messages are not lost since each service implements the outbox pattern.

\paragraph{Keycloak failure.} New logins and token refreshes fail. However, services validate JWTs locally using Keycloak's public key, so requests carrying a still-valid token continue to be accepted until expiry.

\paragraph{Ticketing crash during checkout session creation.} If \texttt{ticketing} crashes after tickets have been reserved and the Stripe session has been created, the session will eventually expire and Stripe delivers a \texttt{checkout.session.expired} webhook allowing \texttt{ticketing} to release the reserved tickets. If the crash occurs before the Stripe session is created, tickets remain stuck in \texttt{PENDING}; a background cleanup job on timed-out pending orders would be needed.

\paragraph{Dual notification path (sync + async).} When an event is created, the \texttt{event} service notifies \texttt{ticketing} synchronously via HTTP and the RabbitMQ message follows as a fault-tolerance fallback.

\paragraph{Consistent backup.} Because each service owns its own database, there is no single point from which to take a globally consistent snapshot. One practical mitigation is to temporarily halt writes across all services, snapshot all databases simultaneously, then resume.
